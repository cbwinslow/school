# 📋 Grading Rubrics

## Overview

This document defines the grading standards for all exercises, lessons, and projects in the curriculum. Use these rubrics to evaluate submissions consistently.

---

## 🎯 General Grading Scale

| Score | Grade | Description |
|-------|-------|-------------|
| 90-100 | A | Excellent - Exceeds expectations |
| 80-89 | B | Good - Meets all requirements |
| 70-79 | C | Satisfactory - Meets basic requirements |
| 60-69 | D | Below Expectations - Needs improvement |
| <60 | F | Failing - Does not meet requirements |

---

## 📝 Exercise Rubric

### Category Weights

| Category | Weight | Description |
|----------|--------|-------------|
| Correctness | 50% | Code produces expected output |
| Code Quality | 25% | Clean, readable, well-structured |
| Style | 15% | Follows language conventions |
| Documentation | 10% | Comments and docstrings |

### Detailed Criteria

#### Correctness (50%)

| Score | Criteria |
|-------|----------|
| 100% | All test cases pass, handles edge cases |
| 80% | Most test cases pass, minor issues |
| 60% | Core functionality works, some failures |
| 40% | Partial functionality, significant issues |
| 0% | Does not compile/run or completely wrong |

#### Code Quality (25%)

| Score | Criteria |
|-------|----------|
| 100% | Excellent structure, DRY, SOLID principles |
| 80% | Good structure, minor improvements possible |
| 60% | Adequate structure, some code smells |
| 40% | Poor structure, difficult to maintain |
| 0% | Unreadable, spaghetti code |

#### Style (15%)

| Score | Criteria |
|-------|----------|
| 100% | Perfect PEP8/ESLint compliance |
| 80% | Minor style issues (1-2 violations) |
| 60% | Several style issues |
| 40% | Many style violations |
| 0% | No style consideration |

#### Documentation (10%)

| Score | Criteria |
|-------|----------|
| 100% | All functions documented, clear comments |
| 80% | Most functions documented |
| 60% | Some documentation |
| 40% | Minimal documentation |
| 0% | No documentation |

## 📊 Competency Assessment Rubric

### Skill Level Definitions

#### Level 1: Beginner (0-40%)
- Can write basic syntax
- Needs guidance for most tasks
- Limited understanding of concepts

#### Level 2: Intermediate (40-70%)
- Can complete guided tasks independently
- Understands core concepts
- Can debug simple issues

#### Level 3: Advanced (70-85%)
- Can design and implement solutions
- Applies best practices
- Can debug complex issues

#### Level 4: Expert (85-95%)
- Can architect systems
- Mentors others
- Contributes to standards

#### Level 5: Master (95-100%)
- Innovates and creates new patterns
- Industry thought leader
- Defines best practices

## 🔄 Re-Submission Policy

### When to Allow Resubmission

- Student scored < 70% and wants to improve
- Student identified and fixed issues
- Learning opportunity exists

### Resubmission Scoring

- Maximum score on resubmission: 90%
- Must address all feedback points
- Must include explanation of changes

### Resubmission Template

```markdown
## Resubmission for {Assignment Name}

### Changes Made
1. {Change 1}
2. {Change 2}
3. {Change 3}

### Issues Addressed
- {Issue 1}: {How it was fixed}
- {Issue 2}: {How it was fixed}

### What I Learned
{Reflection on the learning process}
```

## 📋 Quick Reference

### Pass/Fail Criteria

**Exercise Pass**:
- Score ≥ 60%
- Code compiles/runs
- Core functionality works

**Project Pass**:
- Score ≥ 70%
- All required features implemented
- Documentation present
- Tests written

**Module Pass**:
- All lessons completed
- Average exercise score ≥ 70%
- Project score ≥ 70%
